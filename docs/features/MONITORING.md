# 流量监测功能文档

## 概述

Interface Proxy 提供了简单的内存流量监测功能，用于统计不同模型的请求次数和成功率。

## 功能特性

- ✅ 基于小时级别时间桶的聚合统计
- ✅ 支持 24h/7d/30d 时间范围查询
- ✅ 按模型维度统计请求数、成功数、失败数
- ✅ 极简实现，内存占用 < 1MB
- ✅ 对请求性能影响可忽略（< 0.01ms）
- ✅ 自动清理超过 30 天的旧数据

## 架构设计

### 数据结构

使用小时级别的时间桶存储聚合计数：

```python
buckets = {
    hour_timestamp: {
        model_name: {
            "count": int,     # 总请求数
            "success": int,   # 成功数
            "error": int      # 失败数
        }
    }
}
```

### 内存占用

- 时间桶数量：30天 × 24小时 = 720 个桶
- 每个桶：假设 10 个不同模型
- 每个模型：3 个计数器（count, success, error）
- 总内存占用：**约 500KB - 1MB**（包含字典开销）

### 性能特点

- **记录操作**：O(1)，只递增计数器，< 0.01ms
- **查询操作**：O(桶数 × 模型数)，毫秒级
- **清理操作**：每 100 次记录触发一次，影响可忽略

## API 接口

### 1. 查询统计数据

**接口**：`GET /api/stats`

**参数**：
- `time_range`（可选）：时间范围，支持 `24h`、`7d`、`30d`，默认 `24h`
- `model`（可选）：模型过滤，为空时返回所有模型

**示例**：

```bash
# 查询 24 小时统计
curl "http://127.0.0.1:8080/api/stats?time_range=24h"

# 查询 7 天统计
curl "http://127.0.0.1:8080/api/stats?time_range=7d"

# 查询特定模型
curl "http://127.0.0.1:8080/api/stats?time_range=24h&model=gpt-4"
```

**响应示例**：

```json
{
  "time_range": "24h",
  "total_requests": 150,
  "success_count": 145,
  "error_count": 5,
  "by_model": [
    {
      "model": "gpt-4",
      "count": 100,
      "success": 98,
      "error": 2
    },
    {
      "model": "gpt-3.5-turbo",
      "count": 50,
      "success": 47,
      "error": 3
    }
  ]
}
```

### 2. 查询模型列表

**接口**：`GET /api/stats/models`

**示例**：

```bash
curl "http://127.0.0.1:8080/api/stats/models"
```

**响应示例**：

```json
{
  "models": [
    "gpt-4",
    "gpt-3.5-turbo",
    "claude-3-opus"
  ],
  "count": 3
}
```

## 实现细节

### 统计模块

位置：`proxy_app/monitoring/simple_stats.py`

核心类：`SimpleStatsCollector`

主要方法：
- `record(model, status)`：记录一次请求
- `get_stats(time_range, model)`：查询统计数据
- `_cleanup()`：清理过期数据

### 统计中间件

位置：`proxy_app/app.py`

功能：
1. 拦截 `/v1/chat/completions` 请求
2. 提取模型名称
3. 根据响应状态码判断成功/失败
4. 调用统计收集器记录数据

### 线程安全

使用 `threading.Lock` 保证多线程环境下的数据一致性。

## 使用场景

### 适用场景

- ✅ 单机部署
- ✅ 简单粗略统计
- ✅ 不需要长期历史数据
- ✅ 对数据丢失不敏感（重启后清零）

### 不适用场景

- ❌ 需要持久化存储
- ❌ 多实例部署需要汇总统计
- ❌ 需要分钟级别精度
- ❌ 需要详细的请求日志

## 常见问题

### Q: 重启后数据会丢失吗？

A: 是的，统计数据存储在内存中，服务重启后会清零。如果需要持久化，可以考虑添加定期备份功能。

### Q: 多实例部署如何统计？

A: 当前实现是单机统计，多实例部署时各自独立统计。如需汇总，可以考虑使用 Redis 等外部存储。

### Q: 统计精度是多少？

A: 统计精度为小时级别，即同一小时内的请求会聚合在一起。

### Q: 会影响请求性能吗？

A: 影响极小（< 0.01ms），只是简单的计数器递增操作。
